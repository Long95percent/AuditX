# AuditX 架构审查基线与后续建议

> 记录时间：2026-05-19  
> 状态：同事尚未开始新增实现代码，本文件作为后续代码审查的设计基线。

## 1. 审查目标

后续审查重点不是简单判断代码能否运行，而是判断新增实现是否偏离 `docs/plans` 中已经确定的产品与架构方向。

核心目标：

- 保证审查主链路始终以 `AgentOrchestrator` 为唯一入口。
- 避免规则、API route、前端或临时 service 绕过 Agent 主路由。
- 保证风险、评分、分层、优势和证据都落在统一数据契约上。
- 保证每次审查都可解释、可追踪、可降级、可测试。
- 保证当前阶段面向 5000 份简历压测，而不是过早追求 10 万实时处理。

## 2. 必须遵守的设计基线

### 2.1 Agent 主路由

`docs/plans/README.md` 和 `docs/plans/agent_first_routing_design.md` 明确要求：

- `AgentOrchestrator` 是唯一审查主路由入口。
- 外层业务代码只提交文档、岗位模板、运行配置和可用工具。
- 外层业务代码不直接判断应该走哪个规则或哪个 Agent。
- API route 不直接分发规则、Agent 或工具。

后续如果同事新增以下代码，需要重点标记为架构偏离：

- 在 API route 中直接调用具体规则、具体 Agent 或具体 LLM provider。
- 在 `AuditUseCase` 中继续堆叠大量 if/else 来决定审查路径。
- 让某个 rule engine 成为实际主路由。
- 让前端传入“使用某条规则/某个 Agent”的路由决策。

建议方向：

- 新增 `backend/auditx/agent_core/orchestrator.py` 或同等职责模块。
- `AuditUseCase` 可以保留，但应逐步变成 `AgentOrchestrator` 的薄封装。
- 所有审查步骤应通过 orchestrator 产生可追踪 step。

### 2.2 规则工具化

计划文档明确：规则是工具，不是总路由。

后续审查时应确认：

- 规则输入包含结构化简历、岗位模板和当前 Agent 上下文。
- 规则输出包含命中结果、证据候选、分数影响和解释。
- 规则失败只返回错误状态或 trace step，不中断整份简历审查。
- 每新增一条规则，都有测试证明它不会绕开 `AgentOrchestrator`。

需要警惕的问题：

- 规则直接产出最终风险，未经 evidence 校验。
- 规则层负责决定整个审查流程怎么走。
- 规则失败直接导致 job failed，而不是进入降级/不确定状态。

### 2.3 统一数据契约

当前代码已有 `AuditFinding`、`Evidence`、`AuditResult`，但还不足以支撑计划里的简历审查产品形态。

后续应优先补齐或等价实现以下契约：

- `JobTemplate`：硬性要求、优势词典、权重、风险策略、模板版本。
- `FindingCandidate`：Agent、LLM、规则或工具产出的候选发现。
- `ReviewTrace`：记录 Agent 输入、输出、工具调用、采纳/拒绝原因、降级原因。
- `ScoreBreakdown`：维度分、规则贡献、优势加分、风险扣分、计算明细。
- `CandidateLayer`：最优、次优但有潜力、不建议。
- `ReviewResult`：分数、分层、优势、风险、证据、计算明细、trace。

需要警惕的问题：

- 为前端临时增加零散字段，但后端没有统一领域模型。
- 只返回 `findings`，不返回评分、分层、优势、计算明细和 trace。
- 使用字符串字典到处传递核心审查结果，缺少 Pydantic 类型约束。

### 2.4 正式风险必须经过证据校验

当前已有 `EvidenceValidator`，方向正确。

后续应继续保持：

- Agent 可以提出风险候选，但不能绕过校验直接成为正式风险。
- 未通过校验的候选应进入 rejected 或 pending，并在 trace 中可见。
- 证据可以来自简历原文、本地模拟公司信息文件或工具返回结果。
- 当前 12 天计划内不做真实网页搜索，证据链接优先使用本地文件。

需要警惕的问题：

- LLM 输出直接展示成正式风险。
- 没有证据的风险只靠置信度进入结果。
- 被拒绝候选只统计数量，不记录原因和来源。

### 2.5 岗位模板、评分、分层与 Top N

计划文档明确当前产品是按岗位模板审查简历，而不是泛化文档审计。

后续应确认：

- 同一简历在不同岗位模板下可得到不同分数、优势和风险策略。
- 权重由岗位模板控制，而不是写死在评分函数中。
- 分数不是最终裁决，必须保留维度分和计算明细。
- Top N 支持默认 `N = 20` 和 HR 自定义 N。
- Top N 排序并列规则为风险更少优先、优势更多优先、入库时间更新优先。

需要警惕的问题：

- 简历只有全局分数，没有岗位上下文。
- 某个维度一票否决，但岗位模板没有配置硬性淘汰项。
- 分层逻辑散落在前端或 API 层。

### 2.6 简历库状态与历史复用

计划文档要求支持：新入库、已筛查、已入岗位候选库。

已筛查简历再次进入新岗位或新批次时：

- 可以复用解析结果、结构化信息、历史风险证据和历史 trace。
- 必须重新计算当前岗位下的匹配分。
- 必须重新计算当前批次下的 rank、Top N 和分层。

需要警惕的问题：

- 把历史分数直接用于新岗位。
- 把历史 Top N 或分层结果直接复用到新批次。
- 只有内存 job 字典，无法支撑批量任务、历史复用和重启恢复。

### 2.7 批量任务与 5000 份压测

计划目标是单日 5000 份简历累计处理。

后续新增批量能力时，应关注：

- 是否有批量任务模型和单份处理状态。
- 是否支持失败隔离，单份失败不影响整批。
- 是否支持分页、筛选和结果摘要。
- 是否有压测数据生成器或黄金测试集。
- 是否避免一次性把全部简历和全部结果塞入前端内存。

需要警惕的问题：

- 同步循环处理 5000 份且没有进度状态。
- 批量任务失败后无法定位是哪份简历或哪个步骤失败。
- 没有稳定回归数据，改 Agent 或规则后无法发现分层变化。

## 3. 当前代码状态观察

截至本文件创建时，当前实现属于 Phase 1A fake 闭环：

- `backend/auditx/application/audit_use_case.py` 串联 parser、extractor、evidence validator、normalizer。
- `backend/auditx/agent_core/fake_extractor.py` 产出模拟风险。
- `backend/auditx/tool_registry/registry.py` 已有工具注册表，但尚未接入主流程。
- `backend/auditx/application/audit_job_service.py` 使用内存字典保存 job。
- `frontend/src/types/audit.ts` 只覆盖 job、finding 和 evidence，尚未覆盖评分、分层、优势和 trace。

这些在当前阶段不一定是错误，但如果后续继续沿着 fake extractor 线性扩展，就会偏离计划。

## 4. 后续代码审查清单

每次同事提交后，建议按以下顺序审查：

1. 入口检查：新增审查流程是否统一进入 `AgentOrchestrator`。
2. 路由检查：API、service、前端是否绕过 orchestrator 调用规则或 Agent。
3. 契约检查：新增字段是否进入统一领域模型，而不是临时 dict。
4. 证据检查：正式风险是否全部经过 evidence 校验。
5. Trace 检查：Agent 步骤、工具调用、采纳/拒绝原因是否可见。
6. 降级检查：单个 Agent、规则或工具失败是否不会中断整份审查。
7. 岗位检查：评分、优势、风险是否受岗位模板控制。
8. 批量检查：是否支持单份失败隔离、分页、筛选和压测数据。
9. 测试检查：是否有路由测试、失败隔离测试、evidence 测试和黄金集回归测试。

## 5. 建议的第一批审查门槛

同事开始写代码后，建议第一批 PR 或提交至少满足：

- 有 `AgentOrchestrator` 或等价主路由对象。
- 有 `FindingCandidate` 和 `ReviewTrace` 的最小模型。
- 调用审查流程时一定产生至少一个 Agent step trace。
- 工具或规则失败时主流程不中断，并在 trace 中记录失败。
- API route 只调用 application service，不直接调用具体规则、Agent 或工具。
- 至少新增一条测试验证“规则不会绕开 orchestrator”。

如果第一批实现没有满足这些门槛，建议先暂停扩展评分、前端列表和批量能力，优先把主路由和数据契约补齐。

## 6. 推荐记录格式

后续每次审查建议在 `docs/suggestions` 新增独立文件，文件名格式：

```text
YYYY-MM-DD-<short-topic>.md
```

每份审查记录建议包含：

```markdown
# <审查主题>

## 结论

- 通过 / 有条件通过 / 不建议合入。

## 观察

- 观察到的事实，带文件路径和行号。

## 偏离点

- 对照 `docs/plans` 的具体偏离。

## 风险

- 如果继续这样写，会造成什么后果。

## 建议

- 可执行的修改建议，按优先级排序。

## 建议测试

- 需要补充或运行的测试。
```
