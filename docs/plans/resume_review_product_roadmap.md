# 简历审查效率工具产品路线图

## 1. 总体目标

AuditX 后续聚焦为 **简历审查效率工具**：帮助 HR / 招聘团队更快完成简历初筛、信息完整性检查、经历一致性检查、岗位匹配检查和证据化复核。

产品原则：

- 先提升审查效率，再逐步提升智能程度。
- 先做可解释规则，再引入模型推理。
- 每个结论都尽量可回溯到简历原文或外部证据。
- 架构上保持解析、规则、模型、工具、展示解耦，方便后续扩展到更多招聘场景。

## 2. 阶段路线总览

| 阶段 | 产品目标 | 核心能力 | 验收重点 |
|---|---|---|---|
| Phase 1 | 单份简历快速检查 | PDF 解析、基础规则、证据展示 | 能稳定审查一份中文简历 |
| Phase 2 | 中文简历可用性增强 | OCR、DOCX、更多规则、岗位配置 | 覆盖常见简历格式和岗位要求 |
| Phase 3 | 批量初筛效率工具 | 批量任务、排序、筛选、导出 | HR 可处理一批候选人 |
| Phase 4 | 岗位匹配与评分体系 | JD 解析、匹配分、解释维度 | 从“找问题”升级到“辅助排序” |
| Phase 5 | 可信度与背景辅助核验 | 公司库、外部工具、异常经历识别 | 可辅助发现可疑经历 |
| Phase 6 | 团队协作与企业化 | 多用户、权限、审查流、审计日志 | 支持团队真实使用 |
| Phase 7 | 智能 Agent 增强 | LLM 总结、问答、策略推荐 | 在可控边界内提升智能体验 |

## 3. Phase 1：单份简历快速检查

目标：做出一个能演示、能评审、能闭环的单份简历审查工具。

核心能力：

- 支持选择本地 PDF 简历。
- 解析原生文本 PDF，生成 `ParsedDocument`。
- 运行基础规则：联系方式、教育经历、工作时间重叠、公司名称模糊、工作年限不足。
- 输出风险等级、规则说明、置信度、页码和原文证据。
- 前端展示风险列表和基础文档信息。

暂不做：

- 不做批量处理。
- 不做真实外部核验。
- 不做复杂 LLM 决策。

验收标准：

- 3-5 份样例简历能稳定返回审查结果。
- 正式风险都通过 evidence 校验。
- 解析失败或扫描件有明确提示，不伪造结论。

## 4. Phase 2：中文简历可用性增强

目标：让工具覆盖更多真实中文简历格式。

核心能力：

- 支持 DOCX 解析。
- 接入 OCR，支持扫描件 PDF 和图片简历。
- 增加中文时间表达解析，例如 `2020年3月-2022年6月`、`2021.07 至今`。
- 增加学历、证书、项目经历、技能关键词等规则。
- 支持岗位配置：目标岗位、最低年限、关键词、学历要求、加分项。

产品变化：

- 从“固定规则检查”变成“按岗位要求检查”。
- 用户可以选择岗位模板，例如前端、后端、产品经理、财务、人事。

验收标准：

- 常见 PDF / DOCX / 扫描件都能进入统一审查流程。
- 岗位配置能影响规则阈值和匹配结果。
- OCR 低置信度内容必须在结果中标记，不作为高置信度结论。

## 5. Phase 3：批量初筛效率工具

目标：从单份审查升级为批量初筛，真正提升 HR 工作效率。

核心能力：

- 批量导入文件夹或多份简历。
- 支持从简历库选择本次输入，不要求每次重新上传。
- 标记简历状态：新入库、已筛查、已入岗位候选库。
- 提供预定义筛选：按时间排序、只看新简历。
- 后台任务队列，展示每份简历状态。
- 候选人列表页：姓名、岗位匹配度、风险数、高风险标签、解析状态。
- 支持筛选和排序：高风险优先、匹配度优先、解析失败、缺少联系方式等。
- 导出 CSV / Excel 审查摘要。

产品变化：

- 从“审查工具”变成“初筛工作台”。
- HR 先看列表排序，再进入单份详情复核。
- 已筛查简历可复用解析结果和结构化信息，但当前岗位 / 当前批次的 rank 必须重新计算。

状态流转：新上传 / 导入 -> 新入库 -> 完成审查后已筛查 -> HR 选择后已入岗位候选库。同一简历可以多次参与不同岗位或批次审查。

验收标准：

- 一次处理 50-100 份简历不会阻塞界面。
- 失败任务可重试。
- 导出结果与详情页结果一致。

## 6. Phase 4：岗位匹配与评分体系

目标：从“发现问题”升级到“辅助判断候选人与岗位的匹配度”。

详细评分、分层、优势标签、风险证据和 HR 呈现方案见：`docs/plans/resume_review_scoring_and_presentation_design.md`。

核心能力：

- 支持输入或粘贴 JD。
- 解析 JD 中的硬性要求、加分项、关键词、年限、学历、技能栈。
- 建立评分维度：基础信息、硬性门槛、技能匹配、经历相关性、风险扣分。
- 每个分数给出解释，而不是只给总分。
- 支持不同岗位模板配置评分权重。

产品变化：

- 输出从“风险清单”扩展为“候选人审查报告”。
- HR 可以按匹配分和风险等级快速排序。

验收标准：

- 同一批简历在同一 JD 下可复现排序。
- 分数解释能追溯到简历文本或 JD 条件。
- 硬性不满足项和普通扣分项清晰区分。

### 6.1 岗位模板化是核心产品能力

不同岗位对简历的判断标准差异很大，岗位模板不应只是筛选条件，而应成为匹配度计算、规则启用和排序解释的核心输入。

岗位模板建议包含：

- **硬性门槛**：学历、最低年限、必备证书、必备技能、地域或语言要求。
- **核心能力**：岗位最重要的技能簇，例如前端的 React/Vue、后端的 Java/Spring、财务的税务/审计经验。
- **经历偏好**：行业背景、项目规模、管理经验、B 端 / C 端经验、外企 / 创业公司经验。
- **风险策略**：时间重叠、频繁跳槽、公司名称模糊、空窗期等风险的扣分方式。
- **加分项**：名企经历、相关证书、开源项目、管理经验、特定业务领域经验。
- **权重配置**：不同维度在总分中的占比，以及硬性淘汰线。

同一份简历在不同岗位模板下应产生不同结果。例如：

- 前端岗位重点看技术栈、项目经历、框架匹配、作品或工程经验。
- 财务岗位重点看证书、审计/税务经验、稳定性和行业合规背景。
- 产品岗位重点看行业理解、项目闭环、数据分析和跨部门协作。
- 销售岗位重点看行业客户、业绩数字、区域经验和稳定性。

因此，匹配度结果不能只输出一个总分，至少应拆成：

- 硬性门槛结论：通过 / 不通过 / 待人工确认。
- 技能匹配分：候选人技能与岗位关键词的重合度。
- 经历相关分：行业、项目、职责和岗位要求的相关性。
- 风险扣分：时间线、信息缺失、模糊表达等问题。
- 加分项：超出岗位要求的优势信号。

### 6.2 岗位模板的产品形态

第一版可以内置少量模板，后续再开放配置：

- 技术类：前端、后端、测试、算法、运维。
- 职能类：HR、财务、法务、行政。
- 业务类：销售、运营、市场、产品经理。
- 管理类：项目经理、团队负责人、部门经理。

模板后续应支持从 JD 自动生成初稿，再由 HR 调整权重和硬性要求。这样既能降低配置成本，也能保留人工控制权。

## 7. Phase 5：可信度与背景辅助核验

目标：增强对可疑经历、模糊公司、异常时间线的识别能力。

核心能力：

- 本地公司词库和常见公司白名单。
- 模糊公司名识别，例如“某科技公司”“知名外企”“XXX 集团”。
- 工作经历时间线核验：重叠、空窗期、过短经历、频繁跳槽。
- 可插拔外部工具接口：企业信息查询、学校/证书规则库、行业关键词库。
- 外部工具结果只作为辅助证据，不直接替代人工判断。

产品变化：

- 从“简历内容检查”扩展到“可信度辅助核验”。
- 仍然强调证据和解释，避免黑箱判断。

验收标准：

- 外部工具可 mock，便于测试和演示。
- 无外部网络时系统仍可运行基础规则。
- 外部核验失败时降级为提示，不影响主流程。

## 8. Phase 6：团队协作与企业化

目标：支持招聘团队在真实流程中使用。

核心能力：

- 用户、角色和权限。
- 项目 / 岗位维度管理候选人。
- 审查状态：待审、已通过、待确认、拒绝、已导出。
- 人工备注、标签、复核记录。
- 审计日志：谁在什么时候查看、修改、导出过结果。
- 数据脱敏和本地部署选项。

产品变化：

- 从个人效率工具升级为团队招聘审查系统。
- 开始具备企业采购和私有化部署价值。

验收标准：

- 不同角色只能访问授权岗位和候选人。
- 所有人工决策都有记录。
- 可导出用于复盘的审查记录。

## 9. Phase 7：智能 Agent 增强

目标：在规则和证据体系稳定后，引入可控的智能能力。

Agent 与规则路由应按 `docs/plans/agent_first_routing_design.md` 设计：Agent 编排优先，规则作为工具和校验器逐步沉淀，避免规则路由过早限制 Agent 能力。

核心能力：

- 自动生成候选人摘要。
- 支持对单份简历问答，例如“这个人为什么不匹配？”
- 根据 JD 生成建议面试问题。
- 对风险点生成复核建议，例如“建议询问 2022-2023 时间重叠原因”。
- 支持多 Agent 分工：解析 Agent、规则 Agent、匹配 Agent、核验 Agent、报告 Agent。

约束：

- LLM 不能绕过 evidence 校验直接生成正式风险。
- LLM 输出必须标记来源：原文证据、规则推断、模型建议。
- 高风险结论必须可解释、可复核、可人工覆盖。

验收标准：

- Agent 建议不会破坏规则结果的确定性。
- 所有模型生成内容都有明显标识。
- 用户可以关闭智能增强，只保留规则检查。

## 10. 可扩展性设计

### 10.1 解析层扩展

保持统一输出 `ParsedDocument`：

- PDF parser、DOCX parser、OCR parser 都实现同一解析接口。
- 上层规则和 Agent 不依赖具体解析库。
- block 必须保留 `page_number`、`block_id`、`text`、`bbox`、置信度等信息。

### 10.2 规则层扩展

规则应注册化、配置化：

- 每条规则有 `rule_id`、适用岗位、默认风险等级、触发条件、证据策略。
- 规则可以按岗位模板启用或禁用。
- 规则输出统一为 `AuditFinding`，继续走 evidence 校验和标准化。

### 10.3 岗位模板扩展

岗位模板应独立于代码：

- 模板包含岗位关键词、最低年限、学历要求、证书要求、权重配置。
- 后续可从 YAML / JSON / 数据库读取。
- 不同行业可以新增模板，不影响核心流程。
- 模板应有版本号，保证历史审查结果可追溯到当时使用的岗位标准。
- 模板应支持继承，例如“后端工程师”继承“技术岗通用模板”，再覆盖 Java / Go / Python 等细分要求。
- 模板修改后不应自动改写历史结果，必要时通过重新审查任务生成新版本结果。

### 10.4 工具层扩展

外部工具通过 `ToolRegistry` 接入：

- 企业信息查询、证书核验、学校库、行业关键词库都作为工具。
- 工具只返回事实或候选证据，不直接生成最终结论。
- 工具必须可 mock，保证演示和测试稳定。

### 10.5 存储层扩展

存储应从内存逐步演进：

- Phase 1 可继续使用内存任务。
- Phase 3 开始需要数据库保存任务、候选人、文件、结果、简历状态和岗位候选库记录。
- Phase 6 增加用户、权限、审计日志、导出记录。

简历库需要区分可复用信息和必须重算信息：

- 可复用：文件元信息、解析结果、结构化简历信息、历史风险证据、历史 trace。
- 必须重算：当前岗位模板下的匹配分、当前批次排序、Top N、分层、当前 JD 相关判断。

### 10.6 前端扩展

前端按页面能力分层：

- 单份详情页：风险、证据、原文定位。
- 批量列表页：排序、筛选、状态、导出、简历库输入选择。
- 简历库筛选：按时间排序、只看新简历。
- 岗位配置页：JD、模板、规则权重。
- 团队管理页：用户、权限、审查流。

### 10.7 规模扩展策略

长期 10 万量级可以实现，但它通常指 **10 万份简历 / 候选人记录 / 历史审查结果**，而不是 10 万并发实时审查。当前阶段压测目标先收敛到 **单日 5000 份简历**，产品和架构按批处理系统设计。

核心策略：

- **异步任务**：上传、解析、OCR、规则检查、评分都进入任务队列，前端只查询状态。
- **分阶段落库**：文件元信息、解析结果、风险结果、匹配分、人工复核记录分表保存。
- **结果缓存**：同一文件 hash、同一岗位模板版本下的审查结果可复用，避免重复解析。
- **分页和索引**：候选人列表必须按岗位、批次、状态、分数、风险等级建立索引。
- **批量导入限制**：单次导入可先限制为 500-1000 份，后台分批执行，避免界面和服务阻塞。
- **可恢复任务**：解析失败、OCR 失败、评分失败应独立记录并支持重试。
- **冷热数据分层**：近期招聘批次保留完整可查详情，历史批次可归档或只保留摘要。
- **水平扩展**：解析 worker、OCR worker、评分 worker 可以按队列拆开扩容。

建议阶段能力：

| 阶段 | 可承载规模 | 设计重点 |
|---|---|---|
| Phase 1-2 | 单份到几十份样例 | 正确性、证据链、规则稳定 |
| Phase 3 | 单日 5000 份、单批 500-1000 份 | 队列、任务状态、批量列表、失败重试 |
| Phase 4-5 | 累计 1 万-10 万份 | 数据库索引、缓存、岗位模板版本 |
| Phase 6+ | 企业长期使用 | 权限、审计日志、归档、横向扩展 |

10 万量级下最重要的产品体验不是“每份实时秒出”，而是：批量导入后状态清楚、失败可恢复、列表可筛选、结果可复核、导出可追溯。

## 11. 长期完成态

完善后的 AuditX 应具备以下形态：

- HR 上传一批简历和一个 JD。
- 系统根据岗位模板自动解析、检查、评分、排序。
- HR 先看候选人列表，再进入高价值候选人详情。
- 每个风险、扣分和建议都能看到证据或推理来源。
- 团队可以协作复核、备注、导出和沉淀岗位模板。
- 系统可以承载 10 万量级候选人历史库，并支持按岗位、批次、风险和匹配度快速检索。
- 系统既能本地私有化运行，也能逐步接入企业工具和智能 Agent。
