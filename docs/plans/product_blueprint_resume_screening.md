# AuditX 智能简历筛选产品蓝图

> 这份文档用于统一产品思路、业务买点、功能边界和工程落地方式。后续 AI 搭建代码时，必须先读本文件，再读 `docs/plans/12_day_execution_plan.md`。本文件的目标不是描述一个临时 demo，而是把 AuditX 落成一个可以真实用于 HR 初筛工作的产品方案。

## 1. 产品初衷

AuditX 的初衷是做一个“靠谱可用”的智能简历筛选工具，而不是一个只能演示几条规则的玩具系统。

招聘场景里，HR 和业务负责人面对的真实问题不是“能不能让 AI 读一份简历”，而是：

- 每天有大量简历进入候选池，人工逐份看成本很高。
- 岗位要求经常变化，同一份简历在不同岗位下的价值不同。
- 简历里既有优势，也有风险，不能只给一个粗暴分数。
- HR 需要知道 AI 为什么这么判断，证据在哪里。
- 风险判断必须能回到简历原文、岗位要求或外部/本地证据源。
- 如果 AI 查了资料、做了推理、调用了规则，过程必须可追踪。
- 最终结果要能辅助决策，而不是替代 HR 做黑箱裁决。

因此，AuditX 的产品定位应该是：

**面向 HR / 招聘团队的 Agent 主导型简历初筛工作台。它把简历解析、岗位匹配、风险审查、证据定位、候选人分层和审查 trace 组合成一个可解释、可复核、可批量运行的筛选流程。**

## 2. 核心卖点

AuditX 不应该只强调“AI 筛简历”，这个说法太泛。更好的卖点是：

### 2.1 不是黑箱打分，而是证据驱动的分层筛选

系统不会只输出“85 分”或“推荐 / 不推荐”。它会把候选人分成三层：

- 最优候选：岗位匹配度高，优势明确，风险少或风险可接受。
- 次优但有潜力：存在短板或信息不确定，但优势足够强，值得人工复核。
- 不建议优先：硬性要求弱、风险较高、证据不足或匹配度明显偏低。

每个分层都必须有解释：优势是什么、风险是什么、证据在哪里、哪些地方需要 HR 复核。

### 2.2 风险审查不是简单规则，而是 Agent + 工具 + 证据门禁

系统采用 Agent 主导的审查流程。Agent 负责理解岗位、简历和上下文，决定调用哪些工具。规则只是工具，不是总路由。

例如，Agent 可以先发现一个风险候选：

> “候选人声称 2021-2023 在 A 公司做增长负责人，但简历里缺少具体项目和业绩数字。”

这个候选不能直接变成正式风险。它必须经过 evidence gate：

- 简历原文里是否有相关 quote？
- 对应页码、block、bbox 是否存在？
- 岗位模板是否认为这个风险重要？
- 本地证据库或联网搜索是否提供了补充证据？
- 如果证据不足，应该进入 rejected candidates，而不是正式 findings。

### 2.3 视觉定位让 HR 能快速复核

HR 不应该只看到一段 AI 总结。每个风险和优势都应该能回到原始材料：

- 简历页码。
- 原文 quote。
- bbox 坐标。
- 证据来源类型。
- 本地证据或联网搜索引用。

前端体验上，HR 点击风险后，应该能看到对应原文位置，而不是自己重新在 PDF 里找。

### 2.4 风险数据源可扩展

短期内，风险数据源可以先包括：

- 简历原文。
- 岗位 JD / 岗位模板。
- 本地模拟公司证据库。
- Agent 推理记录。

中期再扩展：

- 联网搜索结果。
- 公司官网或公开资料。
- 新闻、工商、招聘平台公开信息。
- 历史审查记录。

关键原则是：**外部数据必须作为 evidence source 进入系统，而不是直接让 LLM 口头编结论。**

### 2.5 批量筛选和 Top N 服务真实招聘流程

HR 的真实工作不是只看一份简历，而是从很多简历里挑出优先看的候选人。

因此 AuditX 后续必须支持：

- 简历库。
- 当前岗位候选池。
- 当前批次审查。
- 默认 Top N，例如 Top 20。
- HR 可调整 N。
- 未入选候选人仍可查看，不删除、不丢失。
- 批量失败可追踪、可重试。

## 3. 目标用户和使用场景

### 3.1 目标用户

第一阶段目标用户是：

- 中小团队 HR。
- 招聘负责人。
- 创业团队创始人或业务负责人。
- 需要快速筛选大量简历但没有完整 ATS 系统的团队。

这些用户的共同特点是：

- 时间有限。
- 对招聘质量敏感。
- 不希望 AI 黑箱替自己做决定。
- 希望先看到最值得看的候选人。
- 希望风险点能快速定位和复核。

### 3.2 典型场景

一个典型流程应该是：

1. HR 创建或选择岗位模板。
2. HR 导入一份或一批简历。
3. 系统把简历入库，标记为新入库。
4. HR 对当前岗位发起审查。
5. Agent 读取岗位模板、简历内容和上下文。
6. Agent 调用解析、规则、证据、评分等工具。
7. 系统生成候选风险、正式风险、优势标签、评分和 trace。
8. HR 在列表页看到分层结果和 Top N。
9. HR 点击候选人进入详情页。
10. HR 点击风险或优势，定位到简历原文和证据来源。
11. HR 根据系统建议进行人工复核、加入候选池或标记不优先。

## 4. 产品对象模型

后续所有实现都应围绕这些对象展开，避免代码里到处出现临时 dict。

### 4.1 ResumeRecord

表示一份进入系统的简历。

关键字段：

- `resume_id`：系统内部 ID。
- `filename`：原始文件名。
- `source_path`：本地文件路径或导入来源。
- `imported_at`：入库时间。
- `status`：`new`、`reviewed`、`shortlisted`。
- `latest_document_id`：最近一次解析结果。
- `parse_artifact_uri`：解析中间产物位置。

### 4.2 JobTemplate

表示岗位要求和评分策略。

关键字段：

- `template_id`。
- `name`。
- `version`。
- `hard_requirements`。
- `weights`。
- `advantage_dictionary`。
- `risk_strategy`。
- `elimination_rules`，可选，只有明确配置时才允许一票否决。

### 4.3 AuditJob

表示一次审查任务。

关键字段：

- `job_id`。
- `resume_id` 或 `file_path`。
- `template_id`。
- `status`：`pending`、`running`、`completed`、`failed`。
- `document_id`。
- `score_summary`。
- `error`。
- `created_at`。
- `updated_at`。

AuditJob 不应该无限膨胀。大对象应放 artifact。

### 4.4 FindingCandidate

表示 Agent 或工具发现的风险候选。

候选风险不等于正式风险。它可能因为证据不足被拒绝。

关键字段：

- `candidate_id`。
- `rule_id` 或 `source_agent`。
- `title`。
- `description`。
- `risk_level`。
- `confidence`。
- `evidences`。
- `suggestion`。

### 4.5 AuditFinding

表示正式风险。

正式风险必须满足：

- 有证据。
- 证据能定位。
- 通过 evidence gate。
- 能解释对岗位匹配或风险的影响。

### 4.6 Evidence

表示证据。

关键字段：

- `evidence_id`。
- `source_type`：`resume_text`、`job_template`、`local_knowledge`、`web_search`、`agent_reasoning`。
- `quote`。
- `page_number`。
- `block_id`。
- `bbox`。
- `artifact_uri`。
- `confidence`。

### 4.7 ReviewTrace

表示审查过程。

Trace 不是为了好看，而是为了定位问题：

- Agent 调用了什么工具？
- 输入是什么摘要？
- 输出是什么摘要？
- 哪一步失败了？
- 候选风险为什么被接受或拒绝？
- LLM 或联网搜索的原始上下文在哪里？

### 4.8 Artifact

表示大对象文件引用。

大对象包括：

- LLM prompt。
- LLM response。
- 网页搜索结果。
- OCR 文本。
- PDF 解析结果。
- 压测报告。

数据库里只存 artifact 元信息和 URI，不直接存大内容。

## 5. Agent 主导审查流程

AuditX 的核心不是规则引擎，而是 Agent 主流程。

推荐流程：

```text
输入：resume + job_template + review_context
  -> DocumentParser 解析文档
  -> ResumeStructureAgent 提取结构化简历信息
  -> RiskFindingAgent 发现风险候选
  -> RuleTools 补充明确规则候选
  -> EvidenceAgent 查找证据
  -> EvidenceValidator 执行 evidence gate
  -> ScoringEngine 计算维度分和分层
  -> ReportAgent 生成 HR 可读摘要
  -> ReviewTrace 记录全过程
  -> API / 前端展示
```

### 5.1 Agent 和规则的边界

Agent 负责：

- 理解岗位语义。
- 理解简历上下文。
- 发现可能风险。
- 决定调用哪些工具。
- 整理解释。

规则负责：

- 联系方式缺失。
- 学历缺失。
- 年限不足。
- 关键词匹配。
- 时间重叠。
- 明确格式问题。

规则不能负责：

- 绕过 Agent 决定整条流程。
- 直接输出最终结论。
- 在没有证据时生成正式风险。

### 5.2 LLM 的边界

LLM 可以做：

- 结构理解。
- 候选风险发现。
- JD 到岗位模板草案。
- 解释生成。
- 证据摘要。

LLM 不可以做：

- 无证据下直接生成正式风险。
- 编造公司背景。
- 编造候选人经历。
- 覆盖 evidence gate。
- 覆盖评分引擎。

### 5.3 联网搜索的边界

联网搜索是证据源，不是最终裁判。

当系统未来接入联网搜索时，应遵守：

- 查询词必须记录。
- 搜索结果必须保存 artifact。
- 命中的网页片段必须有来源 URL、抓取时间、摘要和 hash。
- LLM 只能基于搜索结果做解释，不能凭空补充。
- 正式风险仍然必须经过 evidence gate。

## 6. 评分和分层

评分不是为了替 HR 做决定，而是为了排序和解释。

建议维度：

1. 完整度：简历信息是否完整。
2. 硬性要求匹配：学历、年限、地点、行业经验等。
3. 能力匹配：技能、项目、方法论。
4. 经验相关度：过往经历和岗位的贴合度。
5. 优势加分：名企、关键项目、增长成果、技术深度等。
6. 风险扣分：信息缺失、时间冲突、证据不足、背景不清晰。

分层建议：

- `best`：高分、优势明确、风险少。
- `potential`：有明显优势，但存在短板或待复核风险。
- `not_recommended`：匹配度低或风险明显。

注意：默认不要一票否决。除非岗位模板明确配置淘汰项，否则学历、年限等短板应影响维度分，而不是直接删除候选人。

## 7. HR 前端体验

前端不是结果 JSON 展示器，而是 HR 工作台。

### 7.1 单份详情页

单份详情页至少应该包括：

- 候选人基础信息。
- 总分和分层。
- 关键优势。
- 主要风险。
- 风险证据定位。
- 计算明细。
- Agent trace。
- rejected candidates。
- 人工复核建议。

### 7.2 列表页

批量阶段需要列表页。

列表页至少包括：

- 候选人姓名 / 文件名。
- 分层。
- 总分。
- 优势标签。
- 高风险数量。
- 待复核标记。
- 审查状态。
- Top N 是否入选。

### 7.3 视觉定位

点击风险时，前端应能显示：

- 原文 quote。
- page。
- block。
- bbox。
- 证据来源。
- 如果是本地证据或网页证据，展示 artifact 摘要。

## 8. 存储方案

为了兼顾“时间不多”和“不要做玩具”，存储采用渐进方案。

### 8.1 当前阶段

当前阶段使用：

- SQLite 保存结构化状态。
- 本地 artifact 目录保存大对象。

不要把所有内容塞进 `audit_jobs.payload`。

建议目录：

```text
.data/
  auditx.sqlite3
  artifacts/
    jobs/
    resumes/
    llm_runs/
    web_search_runs/
    pressure_tests/
```

### 8.2 SQLite 中的核心表

第一阶段至少需要：

- `audit_jobs`：任务状态。
- `job_artifacts`：任务关联的大对象。

第二阶段再加：

- `resumes`。
- `resume_versions`。
- `batches`。
- `batch_items`。
- `findings`。
- `scores`。

### 8.3 Artifact 原则

凡是可能变大的内容，都应该成为 artifact：

- LLM 输入输出。
- 网页搜索结果。
- OCR 全文。
- 解析后的完整 blocks。
- 压测明细。

SQLite 里只存：

- `artifact_uri`。
- `artifact_type`。
- `sha256`。
- `size_bytes`。
- `created_at`。

## 9. 真实业务代码交付标准

AuditX 可以用 MVP 方式逐步测试，但交付代码不能只是临时 demo。

每一天的交付必须满足：

- 有真实 API 或真实服务层代码。
- 有测试证明核心链路可运行。
- 有错误处理。
- 有状态保存。
- 有 trace 或日志帮助定位问题。
- 不把未来一定会替换的 fake 逻辑伪装成完成。
- 如果使用 mock，必须明确 mock 的边界和替换点。

禁止交付：

- 只有前端假数据。
- 只有 worklog 没有代码。
- 只有 hardcode scoring。
- 没有证据的风险结论。
- 路由绕过 AgentOrchestrator。
- 大对象塞进单个 job JSON 导致后续无法扩展。

## 10. 12 天计划的使用方式

AI 后续搭建时必须按 `12_day_execution_plan.md` 推进。

每次任务都要说明：

- 当前执行 Day 几。
- 本次任务属于这个 Day 的哪一条。
- 修改了哪些文件。
- 新增了哪些测试。
- 是否影响后续 Day。
- 哪些内容仍是 mock。
- 哪些内容已经是真实业务代码。

如果 AI 想做计划外内容，必须先说明为什么当前 Day 被阻塞，以及为什么这个内容是解除阻塞所必需。

## 11. 最小可卖版本应该长什么样

一个可拿给真实用户试用的最小版本应该做到：

1. HR 可以选择岗位模板。
2. HR 可以导入简历。
3. 系统可以完成审查。
4. 系统可以输出分层。
5. 系统可以解释优势和风险。
6. 每个风险可以定位到证据。
7. 审查结果重启后仍然存在。
8. HR 可以查看 trace，知道 AI 做了什么。
9. 批量阶段可以挑 Top N。
10. 压测阶段可以证明系统不是只能跑 3 份样例。

这就是 AuditX 的核心买点：

**不是替 HR 做黑箱决定，而是用 Agent 把简历筛选变成可解释、可复核、可批量、可追踪的工作流。**
