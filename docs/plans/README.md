# AuditX 简历审查工具规划入口

当前规划已收敛为 **简历审查效率工具**，不再以泛化审计系统作为近期目标。

## 1. 当前产品方向

AuditX 面向 HR / 招聘团队，核心目标是提升简历初筛效率：

- 按岗位模板审查简历。
- 支持从简历库选择输入，也支持新上传简历。
- 简历有状态标记：新入库、已筛查、已入岗位候选库。
- HR 可用预定义筛选选择输入：按入库时间排序、只看新简历。
- 支持 HR 输入 `N`，系统自动挑选当前岗位 / 当前批次最适合的 `N` 份简历；默认 `N = 20`。
- 将候选人分为最优、次优但有潜力、不建议。
- 展示每份简历的优势、风险、匹配分和证据。
- 保留内部计算过程，HR 可展开查看规则命中、Agent 查找记录和证据来源。
- 当前阶段压测目标为单日 5000 份简历，长期可扩展到 10 万量级历史库。

## 2. 规划文档

建议按以下顺序审查：

1. `12_day_execution_plan.md`
   - 12 天执行排期。
   - 每天任务、测试和产出。
   - 明确 Day 6、Day 11、Day 12 为重点测试和验收窗口。

2. `resume_review_product_roadmap.md`
   - 产品总路线。
   - 阶段目标。
   - 岗位模板化。
   - 规模扩展策略。

3. `resume_review_scoring_and_presentation_design.md`
   - 指标归类。
   - 评分方式。
   - 候选人分层。
   - HR 列表页和详情页呈现。
   - 测试数据和 5000 份压测目标。

4. `agent_first_routing_design.md`
   - Agent 优先的审查路由。
   - 规则工具化。
   - LLM 放权边界。
   - `FindingCandidate`、`ReviewTrace`、降级策略。
   - 防止规则路由绕开 Agent 能力。

## 3. 当前明确架构原则

- `AgentOrchestrator` 是唯一审查主路由入口。
- LLM / Agent 先负责理解、发现、解释，保证产品链路可用。
- 规则作为工具和校验器逐步沉淀，不提前接管主流程。
- 正式风险必须经过证据校验。
- 单个 Agent、规则或工具失败，不应导致整份简历审查失败。
- 岗位模板控制权重、优势词典、硬性要求和风险策略。
- HR 看到的是分层、优势、风险和证据，不是黑箱机器裁决。

## 4. 12 天计划优先级

1. 固化简历库输入筛选、岗位模板、优势词典、评分维度、Top N 挑选和候选人分层。
2. 设计 `AgentOrchestrator`、`FindingCandidate`、`ReviewTrace`。
3. 跑通 Agent 优先的端到端审查链路。
4. 风险证据先使用简历原文和本地模拟公司信息文件；已筛查简历可复用解析结果、结构化信息和历史风险证据，但当前批次 rank、Top N 和分层必须重算。
5. 生成测试简历，覆盖最优、次优但有潜力、不建议三层。
6. 准备单日 5000 份简历的批量处理和压测设计。

## 5. 已删除的旧方向

以下内容已不作为当前规划主线：

- 泛化审计系统定位。
- 只围绕 fake audit demo 的旧执行计划。
- 过早以规则路由主导审查流程。
- 12 天计划内实现真实网页搜索或生产级公司调查。
- 12 天计划内按 10 万份实时处理做压测。
