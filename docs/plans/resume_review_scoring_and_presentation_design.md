# 简历审查评分、分层与呈现设计

## 1. 设计目标

简历审查不应只列举指标，也不应让机器武断淘汰候选人。系统应做的是：

- 把复杂指标归类成 HR 能理解的几个维度。
- 对每个维度给出分析、分数、证据和不确定性。
- 将候选人先分层，再在每层内展示优势、风险和复核重点。
- 保留内部计算过程，HR 需要时可以点开查看。
- 风险点必须能定位到简历原文、规则依据或 Agent 查找证据。

核心原则：**低学历、年限不足、某项弱，不等于整体不通过；它只影响对应维度分数。** 最终呈现应帮助 HR 更快判断，而不是替 HR 做不可解释的裁决。

## 2. 指标归类

把零散指标收敛为 6 个一级维度，每个维度下再挂规则和证据。

| 一级维度 | 说明 | 示例指标 |
|---|---|---|
| 基础完整性 | 简历是否可联系、信息是否完整 | 手机号、邮箱、教育经历、工作经历是否缺失 |
| 岗位硬性要求 | 是否满足岗位最低要求 | 学历、年限、证书、地域、语言 |
| 能力匹配 | 技能和岗位要求的匹配程度 | 技术栈、工具、业务能力、岗位关键词 |
| 经历相关性 | 过往经历与岗位场景是否相关 | 行业、项目规模、职责范围、管理经验 |
| 优势信号 | 候选人值得关注的亮点 | 名企、核心项目、证书、开源、业绩数字 |
| 风险与不确定性 | 需要 HR 复核的问题 | 时间重叠、频繁跳槽、公司模糊、信息冲突、OCR 低置信度 |

这些维度共同形成候选人的解释型画像。系统不应把某个维度一票否决，除非岗位模板明确配置了“硬性淘汰项”。即便触发硬性淘汰，也要保留其它维度分析，方便 HR 判断是否破格考虑。

## 2.1 简历库输入与筛选

产品主流程不是每次都只能上传新文件。系统应维护一个简历库，HR 可以从简历库中选择本次审查输入，也可以继续上传新简历。

每份简历至少需要状态标记：

| 状态 | 含义 | 可复用信息 |
|---|---|---|
| 新入库 | 文件已进入简历库，尚未完成审查 | 文件元信息、入库时间 |
| 已筛查 | 已经过一次或多次审查 | 解析结果、结构化简历信息、历史风险证据、历史 trace |
| 已入岗位候选库 | HR 已选择进入某岗位候选库 | 候选库记录、人工备注、历史审查结果 |

HR 选择输入时，界面应像数据库筛选一样工作。当前先预定义两种筛选：

- **按时间排序**：按入库时间或更新时间排序，方便 HR 从最近简历开始处理。
- **只看新简历**：只选择新入库、未筛查的简历。

后续可以扩展更多筛选，例如按岗位、来源、学历、城市、风险状态、是否已入候选库筛选。

状态流转规则：

```text
新上传 / 导入简历
  -> 新入库
  -> 完成一次审查后变为已筛查
  -> HR 选择进入岗位后变为已入岗位候选库
```

一份简历可以多次参与不同岗位或不同批次审查。`已入岗位候选库` 不表示不能再审查，只表示它已经被某个岗位收纳过。

### 2.2 已筛查简历的复用策略

已筛查简历再次作为输入时，不应所有步骤都重跑。

可复用：

- 文件 hash 和文件元信息。
- 文档解析结果：`ParsedDocument`、页码、block、bbox、OCR 置信度。
- 结构化简历信息：姓名、联系方式、教育、工作经历、项目、技能。
- 历史风险证据：原文 quote、本地公司模拟库证据、Agent 查找记录。
- 历史 trace，用于解释之前如何得出结论。

应重新计算：

- 当前岗位模板下的匹配分。
- 当前批次下的排序和 Top N。
- 当前岗位的优势标签。
- 与当前 JD / 岗位要求相关的硬性门槛判断。
- 任何依赖模板版本、批次上下文、HR 输入 N 的结果。

原因：历史 rank 属于前一批次 / 前一岗位 / 前一模板版本，不能直接拿来复用。系统可以展示历史分数作为参考，但当前批次必须重新排名。

## 3. 评分方式

### 3.1 分数不是结论

建议每个候选人输出：

- 总体匹配分：用于排序。
- 维度分：用于解释为什么排在这里。
- 风险等级：提示需要复核的程度。
- 优势标签：帮助 HR 快速识别兴趣点。
- 分层结果：把候选人放入最优、次优但有潜力、不建议等层级。

### 3.2 维度分建议

初期可以使用 100 分制，但内部保留每个规则的贡献：

| 维度 | 建议权重 | 说明 |
|---|---:|---|
| 岗位硬性要求 | 25 | 不满足会显著降低总分，但不直接删除候选人 |
| 能力匹配 | 25 | 与岗位关键词、技能要求、JD 条件相关 |
| 经历相关性 | 20 | 看项目、行业、职责与岗位场景的贴合度 |
| 基础完整性 | 10 | 信息缺失会影响 HR 联系和判断效率 |
| 优势信号 | 10 | 发现亮点，支持破格关注 |
| 风险与不确定性 | -10 到 0 | 风险作为扣分和复核提示，不掩盖优势 |

权重应由岗位模板控制。比如销售岗位可提高业绩和行业客户权重，财务岗位可提高证书和稳定性权重，技术岗位可提高技能栈和项目经历权重。

### 3.3 内部计算过程

每个分数都应保留计算明细：

```text
候选人总分
  = 岗位硬性要求得分
  + 能力匹配得分
  + 经历相关性得分
  + 基础完整性得分
  + 优势信号加分
  - 风险扣分
```

每一项明细记录：

- 使用的岗位模板版本。
- 命中的规则或 Agent。
- 输入证据：简历原文、JD 条件、本地公司信息文件。
- 原始分、权重、加减分原因。
- 不确定性说明，例如 OCR 低置信度、公司信息仅来自模拟库。

## 4. 候选人分层

建议先分层，再排序。分层比单纯分数更符合 HR 工作方式。

| 层级 | 定义 | 推荐动作 |
|---|---|---|
| 最优 | 匹配度高，硬性要求基本满足，风险少或可解释 | 优先查看，建议进入下一轮 |
| 次优但有潜力 | 有明显优势，但存在短板、风险或部分要求不足 | HR 根据优势标签决定是否复核 |
| 不建议 | 匹配度低、硬性要求明显不足或高风险较多 | 默认后看，但保留详情可查 |

分层逻辑建议：

- 最优：总分高，硬性要求通过，高风险少。
- 次优但有潜力：总分中等或某项短板明显，但优势信号强。
- 不建议：总分低，或关键要求缺失，或风险过多且缺少优势支撑。

注意：低学历候选人如果技能、经历、优势信号很强，可以进入“次优但有潜力”，而不是直接“不建议”。

### 4.1 Top N 自动挑选机制

批量审查时，系统应支持 HR 输入一个数字 `N`，自动从当前岗位 / 当前批次中挑选最适合的 `N` 份简历，再对这 `N` 份准备分层候选池。

推荐机制：

- 系统默认 `N = 20`，用于快速开始审查。
- HR 可以手动输入 `N`，例如从 500 份简历中挑选最适合的 30 份。
- 如果当前输入简历数少于 `N`，则全部进入候选池。
- 如果 HR 输入 `N <= 0` 或非数字，前端提示重新输入，不启动挑选。
- 系统先按岗位模板计算所有简历的匹配分、风险和优势。
- 对已筛查简历，系统优先复用解析和结构化信息，但当前批次 rank 必须重新计算。
- 系统按综合排序挑选 Top N，综合排序不只看总分，还要考虑硬性门槛、优势信号和高风险数量。
- 被选中的 Top N 再分为最优、次优但有潜力、不建议。
- 未进入 Top N 的简历不删除，仍保留在“未入选 / 可查看”列表，HR 可调整 `N` 或筛选条件后重新生成。

Top N 的排序建议：

```text
综合排序分
  = 总体匹配分
  + 优势信号加权
  - 高风险扣分
  - 证据不确定性扣分
```

默认不建议用单个硬条件直接排除候选人。比如学历低于要求时，应降低“岗位硬性要求”维度分，而不是直接从 Top N 中剔除；只有岗位模板明确配置为“硬性淘汰项”时才可排除，并且要在计算明细中标明。

排序稳定性要求：

- 综合排序分相同时，优先高风险更少的候选人。
- 如果仍相同，优先优势信号更多的候选人。
- 如果仍相同，按入库时间倒序排列，较新的简历靠前。

### 4.2 Top N 后的分层

Top N 和分层是两个步骤：

1. **先挑选**：从当前批次中挑出最值得 HR 优先看的 `N` 份。
2. **再分层**：把这 `N` 份分为最优、次优但有潜力、不建议。

这样做的好处是：

- HR 不需要先看全部简历。
- 次优但有潜力的人不会因为单项短板被埋没。
- 有风险但优势很强的候选人可以进入 Top N，并被标记为“需复核”。
- HR 可以通过修改 `N` 控制工作量。

## 5. 优势词典

每个岗位应配置优势词典，用于生成优势标签和加分项。

示例：

| 岗位 | 优势词典 |
|---|---|
| 前端开发 | React、Vue、TypeScript、组件库、性能优化、可视化、大型项目 |
| 后端开发 | Java、Spring、微服务、高并发、Redis、消息队列、数据库优化 |
| 测试 | 自动化测试、接口测试、性能测试、质量体系、CI/CD |
| 产品经理 | 0-1 产品、增长、数据分析、B 端、用户研究、商业化 |
| 财务 | CPA、中级会计、审计、税务、预算、成本控制、上市公司 |
| 销售 | 大客户、渠道、回款、业绩、行业资源、区域市场 |

优势标签在列表页直接展示，例如：

```text
候选人 A｜次优但有潜力｜优势：React / 性能优化 / 大型项目｜风险：学历低于要求
候选人 B｜最优｜优势：Java / 高并发 / 金融行业｜风险：无明显高风险
```

## 6. HR 视图设计

### 6.1 批量列表页

列表页目标是让 HR 快速决定先看谁。

建议字段：

- 输入来源筛选：新上传 / 简历库选择。
- 预定义筛选：按时间排序、只看新简历。
- Top N 输入框：默认值可配置，HR 可手动输入。
- 当前入选数量：例如“已从 500 份中挑选 30 份”。
- 简历状态：新入库、已筛查、已入岗位候选库。
- 候选人姓名。
- 目标岗位。
- 分层：最优 / 次优但有潜力 / 不建议。
- 总体匹配分。
- 优势标签。
- 风险标签和风险数量。
- 解析状态。
- 最近更新时间。

支持筛选：

- 按岗位筛选。
- 按分层筛选。
- 按优势标签筛选。
- 按风险类型筛选。
- 按解析失败 / 待复核筛选。

### 6.2 单份详情页

详情页目标是让 HR 看到“为什么”。

建议结构：

1. 顶部摘要：分层、总分、推荐动作、主要优势、主要风险。
2. 维度卡片：基础完整性、岗位硬性要求、能力匹配、经历相关性、优势信号、风险与不确定性。
3. 风险列表：每个风险显示等级、描述、影响维度、证据入口。
4. 原文证据：点击风险后定位到具体页、具体 block、具体 quote。
5. 计算明细：默认折叠，HR 点击后查看规则命中、权重、加减分过程。
6. Agent 查找记录：展示 Agent 查询了哪些本地证据文件、命中了什么内容、为什么形成该风险。

## 7. 风险点与证据呈现

风险点不能只显示一句结论。建议每个风险包含：

- 风险标题。
- 风险等级。
- 影响维度。
- 风险描述。
- 对分数的影响。
- 证据类型：简历原文 / JD 条件 / 本地公司信息 / Agent 推理过程。
- 证据链接。
- 人工复核建议。

12 天计划内如果公司信息只做模拟，可以使用本地文件链接作为证据链接，例如：

```text
local://evidence/company_mock_db.json#company=某科技公司
local://evidence/company_mock_notes.md#模糊公司名
```

前端展示时可以显示为“本地证据：公司模拟库 / 模糊公司名规则”，点击后展开对应内容或跳转到本地证据详情视图。

## 8. Agent 在风险发现中的角色

Agent 不应直接替代规则系统做最终判断，而应负责“查找、整理、解释”。

Agent 与规则的路由边界、降级策略和 trace 设计见：`docs/plans/agent_first_routing_design.md`。后续应采用 Agent 编排优先、规则工具化的方式，避免规则路由绕开 Agent 能力。

建议分工：

- `ResumeStructureAgent`：识别教育、工作、项目、技能等简历结构。
- `RiskFindingAgent`：根据岗位模板查找风险候选点。
- `EvidenceAgent`：为风险点寻找证据，包括简历原文和本地模拟公司信息。
- `ScoringAgent`：整理规则命中和权重，生成可解释计算明细。
- `ReportAgent`：生成 HR 可读摘要、优势标签和复核建议。

约束：

- Agent 输出的正式风险必须经过 evidence 校验。
- Agent 查找过程要记录输入、命中证据、未命中项和不确定性。
- 对 12 天计划内演示，外部链接先使用本地文件链接，不依赖真实网页搜索。

## 9. 测试数据设计

需要批量生成简历样本，覆盖正常、边界、风险和岗位差异。

### 9.1 样本类型

| 类型 | 目标 |
|---|---|
| 正常匹配样本 | 验证最优层级和高分排序 |
| 单项短板样本 | 验证低学历、年限不足不会一票否决 |
| 潜力样本 | 验证优势强但有短板时进入“次优但有潜力” |
| 高风险样本 | 验证时间重叠、公司模糊、信息冲突 |
| 解析异常样本 | 验证扫描件、OCR 低置信度、格式混乱 |
| 多岗位样本 | 验证同一简历在不同岗位下结果不同 |

### 9.2 生成字段

生成简历时至少覆盖：

- 基础信息：姓名、手机号、邮箱、年龄、城市。
- 教育经历：学校、学历、专业、毕业年份。
- 工作经历：公司、职位、起止时间、职责描述。
- 项目经历：项目名称、规模、技术或业务关键词、成果。
- 技能证书：技能词、证书、语言能力。
- 风险注入：时间重叠、空窗期、模糊公司名、联系方式缺失、学历不足。
- 优势注入：岗位优势词典命中项、业绩数字、名企、核心项目。

### 9.3 黄金测试集

除随机生成外，还应维护一批人工标注的黄金测试集：

- 每个岗位至少 20 份。
- 每份标注预期分层、主要优势、主要风险、关键证据。
- 用于回归测试，防止改规则后排序和风险解释失控。

## 10. 压力测试目标

当前阶段压力测试上限调整为 **单日 5000 份简历**，不再按 10 万级实时处理设计 12 天计划验收。

测试目标：

- 支持单批或多批累计 5000 份简历进入任务队列。
- 列表页分页、筛选、排序可用。
- 失败任务可重试。
- 处理过程有状态：待处理、解析中、审查中、完成、失败。
- 输出结果可导出摘要。

建议压测指标：

| 指标 | 建议目标 |
|---|---|
| 单日处理量 | 5000 份 |
| 单批导入量 | 500-1000 份 |
| 列表分页响应 | 常规筛选下可接受 |
| 失败任务记录 | 100% 可追踪 |
| 结果一致性 | 同一模板版本下重复运行结果一致 |

## 11. 当前 12 天计划建议落地范围

当前 12 天计划不必做完整企业化，只需把“简历库输入 + 分层 + 优势 + 风险证据 + 测试样本”跑通。

建议优先级：

1. 定义评分维度和分层规则。
2. 定义岗位优势词典结构。
3. 在结果中保留计算明细。
4. 风险点支持证据链接，12 天计划内使用本地模拟文件链接。
5. 生成一批可控测试简历，覆盖最优、次优但有潜力、不建议三层。
6. 将压测目标设为 5000 份简历。
