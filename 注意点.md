# 注意点

写 AuditX 代码时，请特别避免以下架构错误：

1. 不要在 HTTP 请求里同步执行耗时审计流程
   - `POST /api/audit-jobs` 这类接口只能创建任务并返回 job_id/状态。
   - OCR、LLM、复杂规则引擎必须放到后台任务、队列或 worker 中执行。
   - 前端通过 `GET /api/audit-jobs/{job_id}` 轮询状态，不要依赖 POST 一次性返回完整结果。

2. 不要让前端随意传任意本机文件路径给后端处理
   - 后端必须校验文件存在、是普通文件、且位于允许目录内。
   - 允许目录通过 `AUDITX_ALLOWED_DOCUMENT_ROOTS` 配置，默认仅适合本地桌面 MVP。
   - 如果未来开放远程访问，必须改成上传文件、文件 token 或 Tauri capability 授权模式。

3. 不要把端口、API 地址、跨域来源散落硬编码
   - 前端统一使用 `frontend/src/config.ts` 读取 `VITE_AUDITX_API_BASE_URL`。
   - 后端端口、允许来源、文件白名单都应从配置读取。
   - 修改启动端口时必须同步更新启动脚本、Tauri 配置、CORS 和前端配置。

4. 不要用进程内内存状态冒充长期可靠存储
   - 当前 API 默认使用 .data/audit_jobs.sqlite3 保存任务快照，不能绕过 repository 直接写内存 dict。
   - 修改任务状态时必须保存 unning 和最终状态，避免崩溃后无法恢复进度。
   - 后续如果接 worker/队列，必须继续复用 repository 契约或做迁移测试。

5. 不要关闭桌面安全配置后进入生产
   - `tauri.conf.json` 中 CSP 为空只适合开发阶段。
   - 生产构建前必须收紧 CSP、文件访问能力和网络访问范围。

6. 改路由必须同步补测试
   - 新增/修改后端 API 时至少补集成测试。
   - 修改任务状态流转时补服务层单测。
   - 不要只靠手工点 UI 验证路由是否正确。
